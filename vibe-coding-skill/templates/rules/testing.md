# 测试规则

> 本文件记录项目已经确认的测试策略与质量门禁。具体要求必须来自项目实际场景。

## 测试层次

- 单元测试范围与覆盖率目标: 待确认
- 集成测试边界与工具: 待确认
- 端到端测试场景与触发条件: 待确认
- 性能或压力测试的适用条件: 待确认

## 回归策略

- 受影响行为的回归检查方式: 待确认
- 对外部契约的变更测试要求: 待确认
- Bug 修复的复现-修复双向验证: 待确认
- 多级 fallback / 降级链路的验证方式: 待确认
- TTL / 缓存 / 过期 / 刷新状态的验证方式: 待确认
- 跨组件组合链路（前端选择 → 后端端点 → 下游能力）的验证方式: 待确认
- 前端 state-changing 请求的认证 / CSRF 完整性检查方式: 待确认

## 测试数据与环境

- 测试数据管理方式: 待确认
- 外部依赖处理（mock、stub、真实服务）: 待确认
- CI 中的测试执行策略: 待确认

## 质量门禁

- 合并前必须通过的检查: 待确认
- 需要独立审查的变更类型: 待确认
- 跳过测试的审批条件: 待确认

## 补充提示

- 如果项目存在 `A -> B -> C` 式 fallback，不仅要验证每一级能工作，还要验证上一级失败时是否真的切到下一级。
- 如果项目依赖缓存、TTL、过期或刷新机制，至少定义有效态、失效态、恢复态三类验证。
- 如果项目规定前端 POST / PUT / PATCH / DELETE 必须经过统一封装层，请在本文件写明检查命令或测试入口。
- Agent 在执行相关 spec 时，必须先读取本文件对应条目；如果条目仍是“待确认”，应先补项目规则或把它作为 verify blocker 显式报告。
- 复盘时尽量给问题贴一个简短失败模式标签，例如：`组合链路缺失`、`时序状态缺失`、`降级路径缺失`、`能力存在但路由错误`、`规则存在但未绑定门禁`、`证据不足以证明结论`。
