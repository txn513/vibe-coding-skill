## 意图 (Intent)

**要解决什么问题？为谁解决？**

Vibe Coding 核心 Skill 的所有交互（init、spec、plan、prompt、review、retro、doctor 等）都在单 Skill、单会话里完成。Core Rule #5 要求"高风险分离角色必须使用不同身份"，但当前没有任何机制让 Reviewer 真正"独立"——同一会话里做实现又做审查，违反独立性。

**重构目标**: 把 Reviewer 这一独立身份从核心 Skill 中抽出，由 monorepo 兄弟 Skill `vibe-coding-reviewer` 承担；核心按需把兄弟链接到 `~/.codex/skills/`，用户无需感知辅助 Skill 的存在。

**当前问题**: 
- 核心 Skill 的 `vibe review` 只生成上下文文件，提示"发送给独立的 Review Agent"——但用户没有自动化的"按需拉起辅助"路径
- 辅助 Skill 散落在 monorepo 兄弟目录里，没有 symlink，Codex 看不到
- Reviewer 包装脚本硬编码 4 个候选路径，无环境变量兜底，非标安装位置会失败

**预期改善**:
- 用户执行 `vibe review` 后，新会话可自动加载 `vibe-coding-reviewer` Skill
- 用户的 `vibe doctor` 主动告警"辅助未安装"，并给出 `vibe install-auxiliary --all` 修复命令
- 辅助脚本接受 `VIBE_SKILL_ROOT` 环境变量，与 `vibe-cli` 解析方式一致
- 核心 Skill 不新增任何业务规则——`auxiliary-onboarding` 是纯治理基础设施

## 成功标准

- `python3 vibe-coding-skill/scripts/vibe.py install-auxiliary --all` 把 `vibe-coding-reviewer` 链接到 `~/.codex/skills/`
- `vibe doctor` 在未安装辅助时输出警告，已安装时静默
- `vibe review` 生成的提示中明确出现"Vibe Coding Reviewer"字样
- `vibe-coding-reviewer/scripts/review_decision.py` 在新 Codex 会话中可通过 `VIBE_SKILL_ROOT` 找到核心
- 核心 Skill 行数从 171 增至 ≤ 200（仅追加 *Suite Architecture* 段，控制在 10 行内）
- 65 个测试全过；新增 11+ 测试覆盖辅助安装路径

## 约束 (Constraints)

### 技术约束
- 核心 Skill 不接受任何业务规则；新增的 `install-auxiliary` 必须是纯治理命令
- 辅助 Skill 的 `SKILL.md` 不复制核心工作流；只描述独立身份角色与薄包装脚本
- 辅助脚本必须是薄包装，逻辑调用核心；不允许在辅助侧重新实现 `record_review` 等
- 链接方式优先 symlink 而非 copy，monorepo 修改可立即生效

### 业务约束
- 用户安装流程：clone suite → 一次性 `vibe install-auxiliary --all` → 之后 Codex 描述匹配自动激活
- 用户不应被要求手动管理辅助 Skill

### 明确不做什么 (Out of Scope)
- 不为本次实现自动按需安装（即不会在 `vibe review` 时自动 `install-auxiliary`）；doctor 提示即可
- 不引入第二个辅助 Skill；本 Spec 只做"套件基础设施"，不下沉业务
- 不重命名 `vibe-coding-skill/` 为 `vibe-coding/`；命名不一致暂时接受
- 不为辅助 Skill 添加独立的 evidence/retro 流程；它们仍走 Builder 项目的同一份 `.agents/`

## 验收标准 (Acceptance Criteria)

### 正常路径
1. `vibe install-auxiliary --all` 在 monorepo 根运行后，`~/.codex/skills/vibe-coding-reviewer` 是指向 monorepo 兄弟目录的 symlink
2. `vibe review` 输出包含 "Vibe Coding Reviewer" 字样与 `install-auxiliary --all` 修复命令
3. `vibe doctor` 在 `~/.codex/skills/vibe-coding-reviewer` 缺失时输出 Warning，否则静默
4. `vibe-coding-reviewer/scripts/review_decision.py` 通过 `VIBE_SKILL_ROOT=...` 显式指向核心时，能成功调用 `vibe.py review-decision` 并落库

### 边界情况
- 目标位置已是正确 symlink：返回"已正确链接"，不重复创建
- 目标位置是真实目录且无 `--force`：返回警告不覆盖
- 目标位置是真实目录且有 `--force`：删除后重建 symlink
- 套件根目录无 `vibe-coding-skill/SKILL.md`：`_suite_root_from_here` 返回 None，提示用户提供 `--suite-root`

### 错误处理
- 辅助目录缺 `SKILL.md`：抛 `FileNotFoundError`，打印目录路径
- 已安装副本运行 `install_auxiliary.py --list` 且无法推断 suite 根：打印明确错误而非栈追踪

## 非功能需求 (NFR)

### 性能
- `install-auxiliary` 必须在 ≤ 2 秒完成单次 symlink（含 Python 启动）
- `vibe doctor` 增加的辅助检测必须在 ≤ 50ms 内完成

### 安全
- `--force` 删除目标前必须先确认是真实目录或 symlink（不允许跟随 symlink 误删真实数据）；当前实现用 `shutil.rmtree(target)` 仅在 `target.is_dir()` 为真时执行，可接受
- 不接受从环境变量读取的、未经验证的 `VIBE_SKILL_ROOT` 路径；只用作"路径字符串 + 存在性检查"

### 可访问性 / 兼容性
- 脚本须在 Python 3.10+ 全部标准库下工作；无第三方依赖
- 路径解析使用 `pathlib` 与 `os.path.expanduser`；macOS / Linux 行为一致
