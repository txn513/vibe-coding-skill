## 意图 (Intent)

**要解决什么问题？为谁解决？**

Vibe Coding 核心 Skill 的 Core Rule #10 与 #11 对 bug 修复有严格要求：
- #10: bug 修复必须有"修复前可复现"+"修复后不复现"的有序证据
- #11: 影响现有行为的改动必须有回归覆盖

但当前 Builder 角色**自己**写代码、**自己**"验证"、**自己** 录 evidence——没有独立身份验证。同一会话里跑测试写 `verify: passed` 是 Core Rule #5 明确禁止的（"高风险分离角色必须使用不同身份"）。

**重构目标**: 引入第二个辅助 Skill `vibe-coding-debugger`，由独立会话、只读身份承担 bug 复现/回归证据落库。Builder 不能在写完代码后立刻说"已验证"；Debug 必须由别人来做。

**当前问题**:
- bug 修复的 evidence 链不可信——自己写测试自己跑自己录
- 没有"独立身份"的强制落地路径
- `record_evidence.py` 的 `--purpose reproduction | fix-regression` 已经存在但很少用

**预期改善**:
- bug 类型的 Spec 在 Builder 落地代码后，**必须**经过 Debugger 角色的独立复现才能推进到 released
- Debugger 不能修改代码，只能运行现有命令、读 spec、录 evidence
- 复现命令必须真实执行（`--command` 提供 argv），不是"看起来跑了"

## 成功标准

- `vibe-coding-debugger/` 目录有 SKILL.md（约 70 行）、`scripts/record_reproduction.py`（≈ 30 行）、`agents/openai.yaml`（4 行）
- `vibe install-auxiliary --all` 把 `vibe-coding-debugger` 链接到 `~/.codex/skills/`
- `vibe doctor` 检测到 debugger 缺失时输出 Warning
- 包装脚本 `record_reproduction.py` 调用 `vibe.py evidence ... verify --purpose reproduction | fix-regression`，且 `--command` 必传
- 既有 80 个测试全过；新增 ≥ 4 个测试覆盖：spec 内容、wrapper 签名、链接可达、--command 必传

## 约束 (Constraints)

### 技术约束
- 辅助脚本是薄包装；不重新实现 evidence 落库逻辑
- 辅助不持有任何业务规则或项目特定上下文
- Debugger 的 SKILL.md 必须明确禁止写代码

### 业务约束
- Debugger 只读代码；任何写操作都应被工作流本身拒绝（角色分离门禁）
- 复现命令必须真实执行并落库 exit code + output snapshot

### 明确不做什么 (Out of Scope)
- 不改 core 的 `record_evidence.py`；只是更高层的包装
- 不强制 bug Spec 必须走 Debugger 流程（这一步是软性建议，强制会破坏向后兼容）
- 不实现自动找复现命令；这需要 LLM 理解代码，超出治理范畴

## 验收标准 (Acceptance Criteria)

### 正常路径
1. `vibe-coding-debugger/SKILL.md` 内容与 reviewer 的 SKILL.md 同构（同样的章节、同样的薄包装定位）
2. `python3 vibe-coding-debugger/scripts/record_reproduction.py --help` 显示预期签名
3. 该脚本缺 `--command` 时返回非零退出码
4. `vibe install-auxiliary --all` 把 debugger 链接到 `~/.codex/skills/vibe-coding-debugger`
5. `vibe doctor` 在 debugger 缺失时输出 Warning

### 边界情况
- `--command` 包含空字符串：拒绝
- 找不到核心 Skill：打印清晰错误，exit code 非零

### 错误处理
- 调用 `vibe.py evidence` 失败：原始错误透传
- 项目根目录无效：`validate_artifact_name` 之前的项目根检查拒绝
