# cli-sync — 回顾

> 上线日期: 2026-06-13 | 回顾日期: 2026-06-13 20:25 UTC | 参与者: codex, codex-review

## 目标回顾

**最初意图**: vibe-cli 目前只封装了早期 6 个命令（init、spec、specs、plan、check、review），

**实际交付**: 将 `vibe-cli` 扩展为统一入口，新增并接通 `status`、`next`、`doctor`、
`migrate`、`prompt`、`advance`、`evidence`、`review-decision`、`amend`、`risk`、
`intent`、`rule-status`、`policy-scan`、`policy-conflict-add`、`policy-conflict-resolve`、
`boundary` 等命令；同时更新 README，并用真实命令记录 verify 证据与 review 结论。

**差异分析**: 核心目标已完成。没有去改 `vibe-coding-skill/` 本体，也没有新增第二套
CLI 逻辑，而是收敛成对 `scripts/vibe.py` 的薄转发层。额外投入主要花在流程验证上，
包括证据门禁、审查门禁和回顾沉淀，这部分属于治理要求，不是功能蔓延。

## 做对了什么

- 明确守住了边界：只改 `vibe-cli`，不把工作流逻辑复制到 CLI。
- 用真实命令写 verify 证据，而不是只靠口头说明。
- 在已有项目上顺着 Skill 的门禁推进，最后拿到了 `review -> done` 的完整闭环。

## 做错了什么

- 第一次 verify 断言写得太死，导致证据被记录成 `failed`，多走了一次修正流程。
- 并行读取 review/evidence 文件时，出现了“像是没写成功”的假象，增加了排查成本。
- 在空目录验证里一开始用了相对路径，忽略了 `cwd` 变化对 CLI 调用路径的影响。

## Agent 表现评估

### 实现 Agent
- **擅长**: 能把治理约束翻成具体执行顺序，并把状态推进、证据、审查串起来。
- **反复出错**: 在验证脚本上容易默认输出内容和执行路径稳定，导致断言过于乐观。
- **需要补充的规则**: 对 CLI/自动化验证，优先断言稳定的结构信号；跨目录执行时优先用绝对路径。

### Review Agent
- **发现的真实问题**: 薄封装边界守住了，没有引入第二套状态机，也没有新增依赖。
- **漏掉的问题**: 没有额外发现新的代码级缺陷，但 review 依赖的上下文在非 Git 工作区下较弱，后续审查信息仍可增强。

## 规格质量

- **哪些约束写对了**: “CLI 只是薄封装层”“不复制业务逻辑到 CLI”“不修改 .agents/ 数据结构” 这几条非常有效。
- **哪些约束漏了**: 可以明确补一条“验证命令需要覆盖跨目录调用场景”，否则容易遗漏路径解析问题。
- **验收标准是否覆盖了所有线上情况**: 覆盖了主要工作流命令和未初始化项目提示，但还没有专门覆盖 `VIBE_SKILL_ROOT` 环境变量路径。

## 上下文质量

- **AGENTS.md 是否准确**: 部分准确，结构描述是对的，但技术栈、格式化、测试框架等仍是待确认状态。
- **规则文件是否有误导**: 没看到直接误导，但对“并行读写下的验证观察”没有明确提醒。
- **Agent 是否理解了项目结构**: 是，能够区分 `vibe-cli/` 与 `vibe-coding-skill/` 的职责边界。
