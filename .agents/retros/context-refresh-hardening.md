# context-refresh-hardening — 回顾

> 上线日期: 2026-06-13 | 回顾日期: 2026-06-13 20:44 UTC | 参与者: codex, codex-review

## 目标回顾

**最初意图**: `self_analyze` 已经提示当前项目的上下文准确性存在改进空间。现有

**实际交付**: 增强了 `refresh_context.py` 的保守检测与字段写回能力：
支持普通字段格式更新，能在没有根级配置文件时基于源码结构识别 Python，
能从项目内容识别 `unittest/pytest` 等测试框架，并在当前仓库上成功把
`AGENTS.md` 更新为 `语言/运行时: Python`、`测试框架: unittest`。

**差异分析**: 核心目标已完成，而且仍然守住了“证据不足就保持待确认”的边界。
没有扩展到自动推断架构约束、安全要求或部署方案，这部分继续保留给人工确认。

## 做对了什么

- 先修了最直接的字段匹配问题，再补无配置文件时的保守推断，顺序是对的。
- 增加了真实回归测试，覆盖了普通字段写回和多子目录 Python 识别。
- 在当前项目上实际跑了一次刷新，确认改动不是只在临时测试目录里成立。

## 做错了什么

- 这次仍然暴露了并行读取下的旧快照问题，验证观察最好继续串行化。
- 项目当前阶段被刷新成 `开发中 (unknown)`，虽然符合现有逻辑，但分支信息仍然偏弱。
- `AGENTS.md` 里还有很多待确认项，说明 refresh_context 只能补一部分，不能替代人工治理。

## Agent 表现评估

### 实现 Agent
- **擅长**: 能把 `self_analyze` 的泛化建议收敛成一个很小、可验证的脚本改进。
- **反复出错**: 在读回新产物时仍然容易受到并行时序噪音干扰。
- **需要补充的规则**: 对上下文刷新类改动，优先验证“当前项目真的更新了哪些字段”，不要只看命令 stdout。

### Review Agent
- **发现的真实问题**: 确认了这次改动没有扩大写入边界，也没有用猜测覆盖人工确认值。
- **漏掉的问题**: 没有新增阻塞问题，但对 `branch unknown` 这类弱信号没有进一步细分。

## 规格质量

- **哪些约束写对了**: “只增强 refresh_context 与测试”“证据不足保持待确认” 这两条约束很有效。
- **哪些约束漏了**: 可以补一条“字段写回需兼容模板中的加粗与普通字段格式”，这样更直接暴露当前类问题。
- **验收标准是否覆盖了所有线上情况**: 覆盖了主要目标；如果继续加强，可以补 `package.json` 在子目录时的框架/测试推断验证。

## 上下文质量

- **AGENTS.md 是否准确**: 比之前更准确了，至少语言、测试框架和阶段状态不再全是占位。
- **规则文件是否有误导**: 没有明显误导，但还没把“refresh 只能保守补全，不能替代人工判断”写成明确规则。
- **Agent 是否理解了项目结构**: 是，能够理解这是一个根目录弱配置、子目录承载主要实现的仓库。
